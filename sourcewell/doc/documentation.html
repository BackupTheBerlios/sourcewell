0. About this manual

This documentation is not up-to-date. It should give more an idea of what SourceWell can than the current status of the project. If you have questions about the current SourceWell version, please contact with the authors.

This manual has been made with the idea of covering as many aspects of SourceWell as possible. If you are a user who's only aim is to install and use the SourceWell system, you may not have to read it as a whole as advanced features for developers are also explained.

SourceWell 1.0.12 is a development version. It is the first step towards the new architecture that will arise with SourceWell 2. There are no feature enhancements towards SourceWell's 1.0.9 version, but some bugs (one big security bug and another memory bug) were fixed. Backwards compatibility with SourceWell 1.0.9 has been mantained. We expect that SourceWell's development will acquire lots of new features with the new architecture.

2. Introduction

SourceWell is a highly configurable, well documented software announcement and retrieval system entirely writen in PHP3 and database independent. It includes user authentication and autherization system (anonymous/user/editor/admin), sessions with and without cookies, high configurability, multilangual support, ease of administration, RDF-type document backend, several graphical statistics, announcing mailing lists, application indexing by sections, installation support and many other useful features.

SourceWell depends on the PHPLib library (version 7.2d). Only if you want to have diary and weekly mailing lists with the announcements, you should also have Mailman installed in your box.

You can see a fully working example of the SourceWell system at BerliOS SourceWell by visiting http://sourcewell.berlios.de. A close look at it will show you what you can do with SourceWell. BerliOS SourceWell has at this moment more than 1500 applications inserted and has been the main reason for making this software.

BerliOS SourceWell is part of the BerliOS project at GMD FOKUS. Please, have a look at http://www.berlios.de for further information. At this moment, BerliOS also has other services: BerliOS Developer (a SourceForge clone), BerliOS News (news about Open Source - in German) and SourceBiz (a directory with Open Source companies, services, products and solutions - in German).

You can always download the latest version of SourceWell at: http://developer.berlios.de/projects/sourcewell

SourceWell Features:

  • Different type of users (nonauthorized users, anonymous user, users, editors and administrators) with different functions
  • Session management with and without cookies
  • Database independent
  • Advanced configurability from a single file
  • Simple, intuitive use of the system
  • Documentation for further development and/or adjustment
  • Comments on applications
  • Through-the-web reviewing and application administration for editors
  • Through-the-web administration of applications, comments and licenses
  • system FAQ and through-the-web administration of it
  • Dynamic order of applications by date (default), importance, urgency or by alphabetical order
  • "true" software counter for apps download, homepage redirections, etc.
  • Multilingual support (to be improved next)
  • Anonymous users can introduce apps and comments
  • The administrator can easily administrate whether there are anonymous users allowed or not
  • Dynamic permission configuration
  • Stable and development branches for applications
  • XML Backend (RDF-document format)
  • Daily and Weekly automatic Newsletters
  • Version history of the applications
  • Links between apps from the same author
  • "intelligent" application validation for editors
  • EMail advice for editors when apps are inserted or updated
  • EMail advice for administrators when new users register
  • Graphical statistics (a lot of them!)
  • Test page (very helpfull when installing the system and something goes wrong)
  • Web browser independence
  • Cache avoidance
  • Advanced search
  • Static web page generation
  • Intelligent Email insertion
  • Temporary table for pending applications (applications without reviewing)

 

3. Installation

Important note about security:

  • SourceWell's main directory and all its subdirectories are not protected in any way by us! Please make sure that you can put propperly .htaccess files in these directories (especially the include subdirectory). If someone is able to read the local.inc file in the include directory, operator (user name) and password for your database could be known and misused. For further information about .htaccess look at the Apache's documentation.

3.1 Fast installation (for MySQL)

You need to have a webserver (we have tested the system with Apache), PHP3, PHPLib and MySQL installed in your system. Mailman is only needed if you want to use mailing lists.

  • user$ mysql -h host -u database_superuser -p
    (insert superuser_password)
  • mysql> USE mysql;
  • mysql> INSERT INTO user VALUES ( 'host', 'operator', PASSWORD('operator_password'), 'Y', 'Y', 'Y', 'Y', 'Y', 'Y', 'Y', 'N', 'N', 'N', 'N', 'N', 'N', 'N');
  • mysql> FLUSH PRIVILEGES;
  • mysql> CREATE DATABASE sourcewell;
  • mysql> exit;
  • user$ tar -zxvf sourcewell-1.0.12.tar.gz
  • user$ mysql -h host -u operator -p < sourcewell/sql/mysql.sql
    (insert the operator_password)
  • user$ vi sourcewell/include/local.inc
    (insert the correct database parameters; please, keep sourcewell as the database name) 
       var $Host     = "host";
   var $Database = "sourcewell";
   var $User     = "operator";
   var $Password = "password";

Ok, that's it. You've finished installing and configuring SourceWell. Start your browser and visit http://yourserver/sourcewell/index.php3. If you get an error message or you have the impression that the system does not work correctly, have a look at this page http://yourserver/sourcewell/test.php3. It will give you some hints on where SourceWell may be failing.

If everything looks ok, great! As you'll see in the index page, the SourceWell system comes with some inserted example applications and licenses. You can delete or modify them by entering as an administrator. To do so, login as the default administrator with user,editor,admin permissions. At the prompt use the following ID to login (case sensitive): 

        Username: oldfish
	Password: oldfish

Very important: Be sure of inmediately changing the login and password by clicking on the User Admin link. You'll notice this is quite easy to do ;-)

If things do not work, maybe you'll have to read the whole installation chapter :-(.

Future versions of SourceWell will work with GNU gettext and we are looking forward to avoiding this large amount of memory.

3.2. Before starting

You've got to look if your web server is correctly configurated and running. That means that:

  • The web server is capable of recognising and executing PHP3 scripts in user directories
  • PHP is compliled to work with the MySQL database

Please, attend to the respective manuals and documentation for a propper installation.

3.2.1. About PHPLib

For a correct functioning of SourceWell, PHPLib has to be installed. PHPLib is an usefull library which permits SourceWell to have sessions, authentication and in future versions database independence. PHPLib is published under the terms of the GNU Lesser General Public License (LGPL). For more information about PHPLib follow this link: http://phplib.netuse.de/.

SourceWell has been developed with the 7.2d version of PHPLib, which can be downlodaded from http://prdownloads.sourceforge.net/phplib/phplib-7.2d.tar.gz. As SourceWell strictly uses the API given by PHPLib, future versions of PHPLib should also work without problem. It is important to note that version 7.2c of the PHPLib library had a security bug that was fixed in the recommended version. So, if you have got this version installed in your box, please update as soon as possible.

Please read carefully the documentation provided for PHPLib for a successfull installation. The installation of PHPLib is sufficient; you don't need to know more about PHPLib and its functioning for using the SourceWell system.

BerliOS SourceWell entry for the PHPLib library: http://sourcewell.berlios.de/appbyid.php3?id=350.

New versions of SourceWell provide a test script (test.php3) which you can run in order to see if your installation is ok. If not, please follow the described instructions carefully.

3.2.2. About the Database

The next step is to install and configure the database system. SourceWell has been tested with MySQL, so that the following explanations are to be followed with that RDBMS. For other systems, the philosophy will be the same, but you will have to particularize the instructions.

You can download MySQL from: http://www.mysql.com/downloads.

1. Install and configure the RDBMS

2. Create the user SourceWell needs:

  • Login as MySQL's superuser (host should be localhost in the default case)
    user$ mysql -h host -u database_superuser -p
    (insert superuser_password)
  • Choose the mysql database
    (that's where the database users and their privileges are)
    mysql> USE mysql;
  • Insert an operator user for sourcewell, its password and its privileges
    mysql> INSERT INTO user VALUES ( 'host', 'operator', PASSWORD('operator_password'), 'Y', 'Y', 'Y', 'Y', 'Y', 'Y', 'Y', 'N', 'N', 'N', 'N', 'N', 'N', 'N');
  • "Commit" the changes
    mysql> FLUSH PRIVILEGES;

3. Create the data base:

  • Create the data base (with sourcewell as its name; see note below)
    mysql> CREATE DATABASE sourcewell;
  • Exit MySQL
    mysql> exit;

Note: It is important for having a simple configuration of the system to name this database "sourcewell". A different database name would imply editing and modifying many of the default configuration parameters in the local.inc file, which is not recommended.

BerliOS SourceWell entry for the MySQL database: http://sourcewell.berlios.de/appbyid.php3?id=76.

3.4. File extraction

Due to security reasons you should not install or execute SourceWell as root. It's not even necessary.

The installation process is as following:

  1. choose the parent directory where you want sourcewell to be and cd into it
  2. verify that the user public_html exists and is accesible from the web
  3. uncompress SourceWell in the parent directory
    user$ tar -zvxf sourcewell-1.0.12.tar.gz
  4. List the content of the directories. You should have a directory sourcewell with five subdirectories. Read chapter 7 of this documentation for further information about the files.
    user$ ls -R
  5. create the database structures in the database
    user$ mysql -h host -u operator -p < sourcewell/sql/mysql.sql
    (insert the operator_password)

4. Main Functionality

While there are basic activities (looking for applications, viewing statistics, etc.) that are open to everybody, SourceWell provides authorisation for advanced purposes.

Note: In the following, we are going to talk about the default permission configuration of the system (which is also the recommended setting). With SourceWell 1.0a, the permission configuration has been implemented to be changed dinamically, so that the site administrator can easily modify the permissions needed at his will from the include/config.inc file.

Authorisation is based on registration. The SourceWell system has two sets of users: unregistered users and registered users. Between registered users we have different permission levels:

  • anonymous, which is a very special user as we will see later
  • user, which can insert and update applications and write comments on them
  • editor, who is responsible for the application administration
  • admin, who is responsible for the administration of users, sections and categories, comments, licences and the mailing lists.

Permissions are independent and do not have to be unique for a certain registered user. That means that a registered user with only administrator permissions is not allowed to perform actions that are restricted to registered users with editor or user permissions, but a single registered user can have user, editor and administration permissions.

Each level of authorisation comprises certain actions that are listed below. Again, we want the reader to notice, that this is the default configuration. The administrator can change it at his will at the include/config.inc file:

Unregistered users can:

  • Retrieve software applications
  • View the comments on applications
  • Read the FAQ
  • Choose language
  • Login
  • Obtain the forgotten password
  • Become authorised user by entering a form
  • Look at the different licences listed in SourceWell
  • Read the XML Backend
  • Search for applications
  • See applications' developers
  • Join the weekly or daily mailing lists
  • View the advanced graphical statistics

SourceWell has an anonymous user with anonymous permission that is able to:

  • Do everything what unregistered user can do
  • Insert and update applications (inserted applications have a "pending" status until they are reviewed by an editor)
  • Write comments

Registered users with user permission can:

  • Do everything what unregistered user can do
  • Insert and update applications (inserted applications have a "pending" status until they are reviewed by an editor)
  • Write comments
  • Change their user registration data
  • See a list of the system users

Editors (editor permission) are able to:

  • Do everything what unregistered users can do
  • Review pending applications (inserted or updated by registered users)
  • Change application insertions
  • Delete and undelete applications
  • See changed or modified applications (applications that are affected when sections, categories or licenses are deleted or modified by an administrator) and change their status

Administrators (admin permission) are able to:

  • Do everything what unregistered users can do
  • Administrate users and their permissions
  • Administrate (insert, rename and delete) sections
  • Administrate (insert, rename and delete) categories
  • Administrate (insert, rename, change URL and delete) licenses
  • Administrate (modify and delete) comments
  • Check the date integrity of applications in the database
  • View the daily and weekly newsletter
  • Send the daily and weekly newsletter at any time
  • Decide if there can be an anonymous user in the system or not

As explained before, SourceWell's registered users can have several roles. So it is possible (and recommendable) that editors (editor) have also user permissions and that administrators (admin) have also user and editor permissions. This becomes even more important if you personalize the permissions of your system in the include/config.inc file as you can only choose one permission type at a time (whether user, editor, admin or all) for every type of functionality.

If the system administrator wants, the system can have anonymous users. Consequently, visitors do not have to be registered, but can post new applications, changes and comments. There can only be one anonymous user in the system and this user can only have anonymous permission; if you don't want to have one, just delete the user with anonymous permission in the admuser.php3 file (there you can also change its username and its password if you want). The SourceWell system automatically shows an information box at the login screen announcing that the visitor can login as anonymous if an user with anonymous permission exists.

 

5. International support

SourceWell can be easily translated into different languages. If you see that SourceWell does not have support in your language, you're gladly invited to help us with the internationalization of SourceWell by sending us your translation.

You don't need to have any computer or programming experience to do a translation. Keep on reading and you'll find out how easy it is.

Note: future versions will have i18n support through gettext and i12n through po files. We are on it.

5.1. Main outputs

Download the lang-translate.inc file (it also comes in SourceWell's 1.0.12 tarball) . If you edit it, you'll find lines like this: 

     case "Home": $tmp = ""; break;

We will explain it briefly: after the case you will see the English text to translate writen in quotes (in our example, the English text is "Home"). Then you'll find a sort of equation. The content of your translation from English into your language should be placed in between these second quotes. For example, in the case you were making a translation into German, this would be the result for this line: 

     case "Home": $tmp = "Heim"; break;

Ok, now that you're an expert, you'll notice that "Home" is translated into German as "Heim" ;-). The procedure just explained should be repeated with all the lines in this file.

Once you're finished, save it as YourLanguage-lang.inc and please send it to the authors. We will include it in the next releases so that everybody can benefit of your work.

5.2. Translation of the FAQ

It is now donw through a database backend. Every entry in the database contains a field in which the language of the current question and answer is specified.

8. SourceWell's database structure

In the following section, we are going to introduce SourceWell's database structure. It may be interesting for developers to know what is stored in the SourceWell system and how the different tables look like.

For the database structure, please, have a look at the files in the sql subdirectory. We've added to this dump some entries that may serve as example applications and licenses for new users. The dump is intended to work with MySQL.

8.1. Table active_sessions

This table is used internally by PHPLib to keep track of the active sessions. As this library is used at a too low level it is uninteresting for the SourceWell developer.

This table is used to avoid manipulation of the application counters stored in the counter table. SourceWell saves some time the IP address of the visitor for each redirection that makes a counter increase. 8.10. Table software

This table has the actual parameters of applications. Registered users with user permission can modify or change applications through the web.

Any application insertion and modification will also be saved with the name of the user that has done it. We have decided to do so as modifications on a certain application do not have to be done always by the same user (that means that every user can modify all applications). The history table keeps track of the users that did the modifications.

Once modified by a registered user with user permission, the status of the application will change to pending. After being reviewed by an editor, the status will become active. This ensures that incorrect or malicious modifications are not inserted.

Applications modified by editors become automatically an active status (and do not need to be reviewed by the editors themselves). This is what we call "intelligent application validation for editors".

9. For developers

SourceWell is Open Source, so you're invited to contribute to it.

The next paragrahps have been made with the idea of a developer who wants to make further development in SourceWell or who wants to make some modifications so that SourceWell suits his needs.

Note: PHP knowledge is needed for developing or modifying SourceWell. You should also have certain SQL experience if you want to handle with databases.

SourceWell has been thought as an announcement and retrieval system for software, but maybe you're interested in adapting SourceWell for announcing other kind of things. Or maybe you want to improve SourceWell adding features or correcting errors. If one of these is your case, keeping on reading will be of interest for you.

Please report to the authors any adaptions and/or modifications. This could help us knowing what SourceWell should implement in the future and making other SourceWell users benefit from your work.

For advanced uses, PHPLib should also be known. You can read about PHPLib's API in PHPLib's manual. PHPLib is used for sessions, authentication and for the database access. It is very important to understand its functions to achieve a propper comprehension of the SourceWell system.

9.1. Source contribution

If you're planning to contribute source, please read the following information:

  • All configuration data belongs to config.inc. The local.inc file just is the local configuration for the PHPLib library. Please keep it free from other code.
  • Commonly used functions should be added to lib.inc.
  • Functions that refer to statistics belong to statslib.inc.
  • Functions that are used for comments should be placed at cmtlib.inc.
  • Functions that handle boxes have to be inserted in box.inc.
  • Obviously, you're free to use whatever coding style you want. But please try to keep your code as simple as possible.
  • Please, try to write all your outputs in English with the translate function (explained later). This will help to keep the internationalization of this project clear.
  • If you want to be really helpful, write an entry for the ChangeLog.

Important: With SourceWell version 1.0, development has switched to CVS. The following method (anonymous CVS access) is preferred for new developers. The module you wish to check out must be specified as the modulename. When prompted for a password for anonymous, simply press the Enter key:

  • Fetch the current CVS tree over anonymous CVS:
    user$ cvs -d :pserver:anonymous@cvs.berlios.de:/csroot/sourcewell login
    [Password: (just press Enter if you are an anonymous user)]
    user$ cvs -z3 -d:pserver:anonymous@cvs.berlios.de:/cvsroot/sourcewell co modulename
    [Updates from within the module's directory do not need the -d parameter]
  • send us the file you've modified, or send us a diff.

We've also set up CVSWEB on http://developer.berlios.de/cvs/?group_id=23.

Write access to the CVS tree is granted only to developers who have already contributed something useful to SourceWell. If you're interested in that, please contact the authors.

9.2. Main structure of a page

Explaining how most of SourceWell files are structured is important for developers in order to rapidly know what is coded and how it has been done. Afterwards adaptions and modifications are made more easily.

  • Starting (loading) session: 
      page_open(array("sess" => "SourceWell_Session"));

    This is a function provided by PHPLib which allows to have sessions with and without cookies enabled in the visitor's browser. It should be placed at the top of every page and before any other output.

     

  • Authentication

    There are two different types of authentication depending on the page visited.

    Case a: authenticated and not authenticated visitors (both type of users can access the current page) 

        if (isset($auth) && !empty($auth->auth["perm"])) {
      page_close();
      page_open(array("sess" => "SourceWell_Session",
                      "auth" => "SourceWell_Auth",
                      "perm" => "SourceWell_Perm"));
    }

    In the first case, everybody can visit that web page without having logged in. What the code does is looking if the user is authenticated; if it is so, a new session with his parameters is loaded (the default one is closed with the page_close() function given by PHPLib). Loading of sessions is done by means of PHPLib.

    This type of authentication should be the default one. The SourceWell system needs sessions to led PHPLib work with databases that are not in the local host. That's why even if sessions are not necessary for a certain page, they should be inserted to ensure a correct functioning of the system.

    Case b: authenticated users only 

        page_open(array("sess" => "SourceWell_Session",
                    "auth" => "SourceWell_Auth",
                    "perm" => "SourceWell_Perm"));

    This page requires to be logged in. At this point it doesn't matter what permissions the registered user has. If the visitor is not logged in, the login-page (loginform.ihtml) will be automatically called by means of PHPLib's functionality.

    So far, we haven't looked if the page visited requires any special permission. This will be done later in the page and won't be realized by PHPLib; the developer will have to make an if-statement with the required permissions and allow the user to access the content of the page or prompt an error box.

     

  • Header

    The include/header.inc file contains the common functions and initializations almost all SourceWell pages require. What the header inserts in each file is:

       

    • Disabling cache 
        header("Cache-Control: no-cache, must-revalidate");
  header("Pragma: no-cache");

      This disables the browsers to catch our web pages in the local cache. If we wouldn't do cache cleaning, just made modifications may not be shown by the browsers as they may show an earlier version of the webpage with the same URL copied in the cache.

      The first line follows the HTTP/1.1 standard, while the second works under HTTP/1.0.

       

    • Required libraries are included: 
         require("config.inc");
   require("lib.inc");
   require("translation.inc");
   require("lang.inc");
   require("box.inc");

      These are the basic libraries for SourceWell. You can find an explanation of them in the chapter where the files are exposed. The functions they contain are explained later.

       

    • Initializes the translation object 
        $t = new translation($la);

      This is the constructor for the language object. It assures that on every page the object is constructed properly. The value of the $la variable gives the language in which the page will be shown.

      As we pretend to use gettext in the future, this translation object will be removed in the next versions.

       

    • database connection (if required in the page)

      SourceWell works in many pages (and in its menubar) with a database. At this point a persistent connection to the database is build. By using PHPLib's functions we expect to have database independence. 

          $db = new DB_SourceWell;

      Note that for MySQL (SourceWell's only supported database up to the moment), this code is equivalent to build a persistent connection like this: 

          $db = mysql_pconnect($db_host,$db_name,$db_password);

       

    • Head of the page (HTML and BODY tags, metatags, title, css, fonts...)
       
    • top strip and top title

      The top strip is a strip (one line table with 100% width) with some links at the left side and the actual value of the session (if logged in user name, permissions and time to end session and if not logged in the string "not logged in") at the right one. You can personalize the links at the right side in the config.inc file. The actual value of the session is translated into the supported languages.

      The top title is another table (with 100% width) which has two images and the title and slogan of your website. You can configure this propperly in the config.inc file.
       

    • Menubar

      The menubar allows to easily navigate through the SourceWell system. It is placed at the left side of your browser.

      The links shown depend on the permissions of the visitor. If you look at the sections where the files in the main directory are explained, you'll understand what this means. For instance, while the menubar contains a link to "Login" for unauthenticated users, authenticated users (that means users who have logged in) will instead get a "Logout" link.

       

    • Opens table

      The content of each page is exposed in the table just opened. It will be closed in the footer.

     

  • Initializes box objects

    The box objects that are used in the page are constructed at the beginning. Usually you will find two boxes: bx and be. bx is the main box while be is the error message box.

     

  • Content of the page

    • permission if required

      Depending on the page, permissions will be required. For instance, if we want a page only accessible for the editors, we will put this code: 

          if (!$perm->have_perm("editor"))

      In the case shown as an example, any other visitor who hasn't got editor permission will get an error message. This is only an example that is not really true. Instead of editor, you will find a variable such as $config_perm_xxxx (being xxxx the type of functionality, such as administrating sections, administrating the faq, etc.). That means that the permission needed to enter that page is set by this variable. This is done so allowing you to easily change the permission needed just by editing the include/config.inc file and modifying the permission requirements that should be met to access that page.

       

    • functionality of the page

      Here comes the real functionality of the page. For instance, the file for registering as a new user (register.php3) contains here the code for doing a registration.

     

  • Footer

    The include/footer.inc file contains the common ending of almost all SourceWell pages. What the footer inserts in each file is:


    • Close table

      The content table opened in the header.inc file is closed.
       

    • Bottom strip

      The bottom strip (one line table with 100% width) gives the name of the Organisation which owns the website. You can configure it to suit your needs in the config.inc file.
       

    • Ending HTML and BODY tags

     

  • Ending (saving) session

    The PHPLib page_close() function closes (saves) the session for this page. In fact, the SourceWell developer does not have to worry about sessions if he follows the exposed scheme. PHPLib will do the session-job transparently for you. Nice ;-).

This is the main library. It contains all the functions that work with the applications, its parameters and main functionality. This functions are used in almost every SourceWell file.

function mktimestamp($time)

Returns the variable $timestamp. Transforms the time parameter properly into a value accepted by the (MySQL) database.

The use of timestamp in SourceWell's database gets a little clearer when we read the MySQL Manual: "The TIMESTAMP column type provides a type that you can use to automatically mark INSERT or UPDATE operations with the current date and time. If you have multiple TIMESTAMP columns, only the first one is updated automatically."

function timestr($timestamp)

Returns the variable $time. This function transforms the type timestamp used in MySQL into the wanted output.

It prints the date in this format: "%A, %e. %B %Y, %H:%M:%S %Z". An example output could be: "Monday, 1. Jan 2001, 15:46:15 CET".

SourceWell can translate the weekday and the month to the languages supported by the system. In our example, "Monday" and "January" will appear only for the English version.

function timestr_short($timestamp)

Returns the variable $time. This function transforms the type timestamp used in MySQL into the wanted output. This function is used in the top_strip.inc file from the include subdirectory that comes at the beginning of almost all SourceWell files. It prints the date in this format: "%a,%e.%b,%H:%M:%S". An example output would be: "Mon,1.Jan,15:46:15".

SourceWell can translate the weekday and the month to the languages supported by the system. In our example, "Mon" (that stays for Monday) and "Jan" (that stays for January) will appear only for the English version.

function timestr_comment($timestamp)

Returns the variable $time. This function transforms the type timestamp used in MySQL into the wanted output. This function is used to show the creation date of application comments.

9.3.4. Functions in statslib.inc

The statslib library has been thought to develop the statistics for SourceWell. The boxes are implemented internally, so the developer only has to worry about the statistical implementation.

This library has been developed parallel to the box library, so you'll find common (and exclusive) features in them. Future versions may try to make both libraries converge.

function stats($result,$var,$message,$position)

This function returns an ordered list (a sort of top list) for the variable $var. $var gives the type for which the table is shown (app_cnt, sum_cnt, download_cnt, homepage_cnt, rpm_cnt, deb_cnt, tgz_cnt). $result is the database result for a given SQL input. $message is the title of the table. The $position parameter gives the starting position of the applications listed.

It does not require any previous box or table definition as it does this internally. The length of the graphic bar is also internally chosen.

function stats_title($message)

Displays the header of the table with the title given by $message. This function is similar to the box_title($title) function defined in box.inc.

function stats_subtitle($message)

Used to insert subtitles in the table. This function is similar to the box_strip($title) function defined in box.inc.

function stats_display($what,$count,$url,$urlquery,$total)

This function displays the statistics graphically in one row. $what is a string with the name of what is displayed, $count gives the number of times it appears in the database, $url is the corresponding url (only the xxx.php3 url), $urlquery the rest of the URL (made so for achieving session managemente with and without cookies) and $total is the total number of implied applications. In the case $url or $urlquery are "0", the displayed statistics wont have a link.

function stats_end()

Finishes the table. This function is similar to the box_end() function defined in box.inc.

9.3.5. Functions in translation.inc

This library is small, but very important. It has the translate function which allows SourceWell to be supported in several languages.

function translate($phrase)

Returns $tmp. phrase is a parameter that contains the string in SourceWell's native language, English. The returned value, $tmp, will be the string in the translated language. The translate function belongs to the translation object. On every SourceWell page, at the header, the variable $t is an initialization (construction) of this object.

9.4. Supporting translations

The way the translation works is very simple. SourceWell's outputs are written by default in English, but there's an implemented function (the translate($phrase) function) that translates these outputs to other languages. In the include subdirectory, you'll find the files that are responsible for this miracle: the Language-lang.inc. These files contain the English to Language translations and help the translate($phrase) function with its work.

The translation procedure is very simple. Every page in SourceWell has a translation object constructed that is ready for use. This object, whose variable is $t contains a function which will be called for every string that should be translated.

It's better understood with an example. Imagine you would like to have the string Home be able to be translated in other languages other than English. For a normal output, we would write this code: 

      echo "Home";

When wanting to be able to have it translated, we will call the explained translation function of the $t object. Coded: 

      echo $t->translate("Home");

The translation function will see first what language the user has chosen and then return the string in $temp that corresponds to "Home" for that language. That means that if the visitor has chosen the German language, you will get this in your web browser as the output of the upper code: 

      Heim

And that's it. You'll have noticed how simple it is to support internationalization for SourceWell!

 

10. Version numbering policy

SourceWell versions will follow this pattern: a.b.c. While a and b can only be integers, c can be either an integer or a character.

  • a will be increased when the SourceWell system has new, innovative concepts.
    Updating your actual SourceWell system is then not only recommended... it's a must!
     
  • b is increased when inserting major bug fixes, major security fixes, a big amount of further developing or new relevant features.
    Updating your SourceWell system is recommended.
     
  • c is increased when inserting minor bug fixes, minor security fixes, a small amount of further developing or a new language support. c is optional if it is cero. It will become a character in steps of five. That means for example that
    • SourceWell 1.0a = SourceWell 1.0.5 and that
    • SourceWell 2.3c = SourceWell 2.3.15

    Updating your SourceWell system is not required if you do not need the incorporated functionality.

 

11. FAQ

  • When should I use PHP-Nuke and when SourceWell?
    PHP-Nuke and SourceWell have been implemented for two different purposes. While PHP-Nuke is a news system portal, SourceWell is thought to be an announcement and retrieval system, where the announced items (in our case software) are more important than user interaction between themselves. Our recomendation then is: use PHP-Nuke if you want to make your site a meeting point about a certain topic and use SourceWell if you want to announce propperly items.

     

  • Is SourceWell database independent?
    The answer is yes. SourceWell uses PHPLib's functions to access the database, so every database supported by PHPLib should also support the SourceWell system. Up to now, we've only tested SourceWell in BerliOS with MySQL.

     

  • Does SourceWell work with PHP4?
    Theoretically yes. Please report the authors if you have got any problems.

     

  • Can an anonymous user add software and do comments?
    If you wish so, yes; you just have to create an user with anonymous permissions (the default system comes with one anonymous user as you'll quickly find out). There can be only one user with that permission and this user can't have any other permissions.
    If you don't wish to have an anonymous user, just make sure that your system hasn't got any user with anonymous permission in admuser.php3. SourceWell will then only accept software insertions, software modifications and comments by registered users (with user permission).

     

  • Does a mailing list for developers interested in SourceWell exist?
    Yes. Just write it down:     sourcewell-support@lists.berlios.de.

     

  • Why is SourceWell not in my language?
    SourceWell can be easily translated into different languages. If you see that SourceWell does not have support in your language, you're gladly invited to help us with the internationalization. Visit http://sourcewell.berlios.de/html/translating.php3.

     

  • What the hell does importance mean?
    Importance is the sum of all the different counts of an application. That means, that importance adds the number of hits on an app plus the number of downloads (tarballs, RPMs, DEBs and TGZs) and the number of redirected changelogs, CVSs, screenshots and mailing list archives. We think it is an interesting manner of measuring the visitor's interest on applications.

     

  • What is BerliOS?
    BerliOS is the German national OpenSource Center. BerliOS offers web information, documentation, information events, consultancy + further developments, sector-specific solutions, software-solutions + new versions, searchengines and successfull debut in the world of OpenSource Software.
